## What
Hand-write the two role protocols and the symmetric connection objects (spec §4) in `Sources/FoundationModelsACP/Connection/`:

- `Agent` protocol: `initialize`, `newSession`, `loadSession`, `prompt` (returns `PromptResponse` carrying `StopReason`), `cancel` (notification), `authenticate`, `setSessionConfigOption`, deprecated `setSessionMode` (with `@available(*, deprecated)`), plus stabilized session management: `listSessions`, `resumeSession`, `deleteSession`, `closeSession`, `logout`.
- `Client` protocol: `sessionUpdate` (notification), `requestPermission`, `readTextFile`, `writeTextFile`, and terminals: `createTerminal`, `terminalOutput`, `waitForTerminalExit`, `killTerminal`, `releaseTerminal`.
- `AgentSideConnection(stream:) { conn in MyAgent(conn) }` and `ClientSideConnection(stream:) { agent in MyClient(agent) }` — each takes a **factory closure**, not a finished handler, so the handler captures its own connection and can issue reverse-direction calls (without this an `Agent` has no handle to fire `sessionUpdate`/`requestPermission`).
- Dispatch uses the **generated routing table** exclusively — no hand-typed method strings.
- Optional methods are gated by advertised capabilities (`clientCapabilities.terminal`, `fileSystem`, provider hooks); unsupported calls answer JSON-RPC -32601 method-not-found.
- Default protocol implementations for optional methods throw method-not-found so conformers implement only what they support.

## Acceptance Criteria
- [ ] An `Agent` impl wired via `AgentSideConnection` receives `initialize`/`newSession`/`prompt` and can fire `sessionUpdate` + `requestPermission` back through the `conn` handed to its factory
- [ ] A `Client` impl wired via `ClientSideConnection` drives the agent and receives its reverse calls
- [ ] Calling a capability-gated method the peer didn't advertise yields `RequestError` code -32601
- [ ] `setSessionMode` emits a deprecation warning at compile time
